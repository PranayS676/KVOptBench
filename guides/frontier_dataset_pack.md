# Frontier Dataset Pack

This guide defines the recommended public workload pack for testing a self-hosted
frontier or frontier-adjacent model with KVOptBench.

Use this pack after the local mock harness passes and before publishing any real
endpoint claims. The pack is designed to test cache reuse, random-prefix controls,
long-context pressure, retrieval quality, and reproducibility without committing large
datasets to the repository.

## Pack Goals

The first public dataset pack should answer:

- Does prefix/radix caching reduce TTFT on repeated long prefixes?
- Does that improvement disappear on random-prefix controls?
- How does latency grow from 8K to 128K context?
- Does the model retrieve known answers from long contexts?
- Can the result package be reproduced from source URLs, adapter options, and hashes?

The pack is not meant to fully rank all engines or all frontier models. It is the first
credible real-data validation pass.

The first public claim should be narrow:

```text
KVOptBench measures whether shared-prefix cache-aware serving improves latency
compared with random-prefix controls on one self-hosted vLLM or SGLang endpoint.
```

Do not try to compare vLLM and SGLang in the first public result unless both runs use
the same model revision, tokenizer, dataset manifests, prompt template, hardware class,
concurrency, and output settings. A one-engine result with strong controls is more
credible than a broad comparison with hidden confounders.

## Required Inputs

Before running a self-hosted model, prepare:

- selected endpoint config, such as `examples/runpod_vllm_openai_compatible_config.yaml`
- public dataset cache directory
- generated workload directory
- result output directory
- model ID and revision
- engine version and exact server command
- GPU type and count
- dataset manifests generated by the adapters

Recommended local layout:

```text
data/raw/
data/processed/
workloads/generated/
results/raw/
results/processed/
reports/outputs/
```

Do not commit large files under those directories.

## Pack Contents

### 1. QASPER Shared-Prefix Cache Pack

Purpose:

```text
Measure warm-prefix TTFT improvement for repeated long document prompts.
```

Dataset:

```text
QASPER
https://huggingface.co/datasets/allenai/qasper
```

Command:

```bash
kvoptbench dataset prepare \
  --source qasper \
  --mode shared_prefix \
  --download \
  --cache-dir data/raw \
  --split validation \
  --target-input-tokens 32768 \
  --target-output-tokens 256 \
  --max-items 100 \
  --out workloads/generated/qasper_shared_prefix_32k.jsonl \
  --manifest workloads/generated/qasper_shared_prefix_manifest.json
```

Required properties:

- at least 5 papers with multiple questions each
- one `prefix_group_id` per paper
- stable question ordering
- answer provenance stored in metadata
- skipped rows recorded in manifest

Recommended real endpoint run:

```bash
kvoptbench cache-plan \
  --plan-dir configs/qasper_cache_plan \
  --experiment-prefix qasper_cache \
  --provider runpod \
  --engine vllm \
  --model-id your/model \
  --base-url https://your-endpoint/v1 \
  --shared-workload-file workloads/generated/qasper_shared_prefix_32k.jsonl \
  --random-workload-file workloads/generated/qasper_random_prefix_32k.jsonl \
  --output-dir results/raw
```

Expected signal:

- warm shared-prefix TTFT should be lower than cold shared-prefix TTFT when backend
  prefix/radix caching is effective
- random-prefix control should show smaller or no warm benefit

### 2. QASPER Random-Prefix Control Pack

Purpose:

```text
Prove cache gains are due to prefix reuse rather than scheduler noise.
```

Command:

```bash
kvoptbench dataset prepare \
  --source qasper \
  --mode random_prefix \
  --download \
  --cache-dir data/raw \
  --split validation \
  --target-input-tokens 32768 \
  --target-output-tokens 256 \
  --max-items 100 \
  --out workloads/generated/qasper_random_prefix_32k.jsonl \
  --manifest workloads/generated/qasper_random_prefix_manifest.json
```

Required properties:

- similar token buckets to shared-prefix pack
- no intentional prefix reuse
- document/question source IDs recorded
- same evaluator family as shared-prefix pack

Expected signal:

- cache-on and warm pass should not improve random-prefix prompts as much as shared-prefix prompts
- if both improve equally, the result is not strong evidence for prefix reuse

### 3. Project Gutenberg Needle Pack

Purpose:

```text
Measure long-context latency, memory pressure, and known-answer retrieval.
```

Dataset:

```text
Project Gutenberg
https://www.gutenberg.org/
```

Command:

```bash
kvoptbench dataset prepare \
  --source gutenberg \
  --mode needle \
  --download \
  --cache-dir data/raw \
  --book-ids 1342,84,2701 \
  --context-buckets 8192,32768,65536,131072 \
  --target-output-tokens 128 \
  --seed 7 \
  --out workloads/generated/gutenberg_needle_8k_128k.jsonl \
  --manifest workloads/generated/gutenberg_needle_manifest.json
```

Required properties:

- beginning, middle, end, and multi-needle positions
- deterministic needle IDs and expected answers
- context bucket recorded per row
- book ID and rights note recorded per row

Recommended real endpoint run:

```bash
kvoptbench long-context-plan \
  --plan-dir configs/gutenberg_long_context_plan \
  --experiment-prefix gutenberg_long_context \
  --provider runpod \
  --engine vllm \
  --model-id your/model \
  --base-url https://your-endpoint/v1 \
  --workload-file workloads/generated/gutenberg_needle_8k_128k.jsonl \
  --output-dir results/raw

kvoptbench long-context-run --plan-dir configs/gutenberg_long_context_plan
kvoptbench summarize --input results/raw --output results/summary.csv
kvoptbench long-context-compare --input results/raw --output results/long_context.csv
```

Expected signal:

- TTFT and E2E latency should grow with context bucket
- failures or timeouts should be recorded as long-context pressure
- quality should reveal whether the model retrieves the needle across positions

### 4. LongBench Core Subset

Purpose:

```text
Add real long-context QA and retrieval tasks beyond synthetic needles.
```

Dataset:

```text
LongBench
https://huggingface.co/datasets/zai-org/LongBench
```

Recommended first subsets:

```text
qasper
multifieldqa_en
hotpotqa
passage_retrieval_en
repobench-p
```

Command:

```bash
kvoptbench dataset prepare \
  --source longbench \
  --mode long_context_qa \
  --download \
  --cache-dir data/raw \
  --subset qasper,multifieldqa_en,hotpotqa,passage_retrieval_en,repobench-p \
  --max-items 200 \
  --out workloads/generated/longbench_core.jsonl \
  --manifest workloads/generated/longbench_core_manifest.json
```

Required properties:

- subtask preserved in metadata
- expected answers preserved
- context length and original ID recorded
- prompt template version recorded

Expected signal:

- validates long-context behavior on established benchmark tasks
- helps separate synthetic needle success from broader long-context quality

### 5. Small Public RAG Pack

Purpose:

```text
Measure grounded answer behavior and citation-oriented quality.
```

Implemented primary option:

```text
BEIR SciFact
https://huggingface.co/datasets/BeIR/scifact
```

Command:

```bash
kvoptbench dataset prepare \
  --source beir_scifact \
  --mode rag \
  --download \
  --cache-dir data/raw \
  --max-items 100 \
  --out workloads/generated/beir_scifact_rag.jsonl \
  --manifest workloads/generated/beir_scifact_rag_manifest.json
```

Required properties:

- query ID and document IDs recorded
- qrels or expected evidence recorded
- prompt requires answer from context only
- citations or source references requested in output

Expected signal:

- validates RAG prompt shape and basic faithfulness fields
- provides a future path for citation quality scoring

Natural Questions is a recommended future RAG adapter target when a larger Wikipedia QA
corpus is needed.

### 6. BFCL Tool-Calling Pack

Purpose:

```text
Validate structured tool-call prompt shape and placeholder quality fields.
```

Dataset:

```text
BFCL
https://huggingface.co/datasets/gorilla-llm/Berkeley-Function-Calling-Leaderboard
```

Command:

```bash
kvoptbench dataset prepare \
  --source bfcl \
  --mode tool_calling \
  --download \
  --cache-dir data/raw \
  --subset BFCL_v3_simple \
  --max-items 100 \
  --out workloads/generated/bfcl_tool_calling.jsonl \
  --manifest workloads/generated/bfcl_tool_calling_manifest.json
```

Required properties:

- tool list and user request recorded in the prompt
- expected tool name recorded in `expected_answer`
- `expected_schema` records the placeholder JSON tool-call shape
- source file and tool count recorded in metadata

Expected signal:

- validates that tool-calling requests execute through the same runner path
- provides a future path for richer BFCL scoring

## Minimum First Run

If GPU time is limited, run only:

1. `qasper_shared_prefix_8k.jsonl`
2. `qasper_random_prefix_8k.jsonl`
3. `qasper_shared_prefix_32k.jsonl`
4. `qasper_random_prefix_32k.jsonl`

That is enough to test:

- cache warm/cold behavior
- random-prefix control behavior
- cache miss penalty
- quality on answerable paper QA
- end-to-end result packaging

Recommended first run shape:

```text
dataset: QASPER validation split
engine: one of vLLM or SGLang
model: one self-hosted frontier-adjacent model
context buckets: 8K and 32K
concurrency: 1, 4, 8
passes: cold and warm
items: 50-100 shared-prefix and 50-100 random-prefix
```

Add `gutenberg_needle_8k_128k.jsonl` after the cache run succeeds. That becomes the
first long-context pressure add-on, not a blocker for the first cache claim.

## Full First Run

If the endpoint is stable and GPU budget allows, run:

1. QASPER shared-prefix cache pack
2. QASPER random-prefix control pack
3. QASPER prefix overlap sweep
4. Gutenberg needle pack
5. LongBench core subset
6. BEIR SciFact small RAG pack
7. BFCL tool-calling pack
8. Strategy Advisor over all comparison CSVs
9. public result template
10. blog-ready report template

## Result Package Structure

The first public result package should include:

```text
README_result.md
run_manifest.json
dataset_manifest_shared.json
dataset_manifest_random.json
workload_sample_10_rows.jsonl
raw_results.jsonl
summary.csv
cache_comparison.csv
report.md
plots/
server_command.txt
config.yaml
version_snapshot.txt
missing_metrics.json
known_limitations.md
```

Trust markers:

- exact model revision
- exact engine version
- exact workload hash
- exact prompt template hash
- tokenizer ID and tokenizer revision
- cold/warm run order
- randomization seed
- request order
- streaming mode
- timeout policy
- retry policy
- failed-request count
- missing metrics

Do not filter failed requests out of public result packages. Failures are part of the
serving behavior being measured.

## Interpretation Rules

Cache result is credible only when:

- shared-prefix warm pass improves compared with cold pass
- random-prefix control does not improve similarly
- input token buckets are comparable
- `prefix_hash` proves the shared-prefix requests actually share text
- backend cache strategy is documented
- missing cache telemetry is listed if unavailable
- engine-reported cache metrics are separated from KVOptBench timing proxies

Long-context result is credible only when:

- context buckets are recorded
- failure/timeouts are preserved
- quality fields are present
- GPU memory telemetry is recorded or listed as missing
- prompt truncation is recorded

RAG result is credible only when:

- corpus source is recorded
- document IDs are recorded
- retrieval/qrel metadata is recorded
- unsupported citation metrics are listed as missing or placeholder

Direct comparison is not credible when model revision, engine version, tokenizer,
prompt template, context bucket, strategy flags, hardware, sampling parameters, request
order, timeout policy, or retry policy differ. Label those comparisons exploratory.
